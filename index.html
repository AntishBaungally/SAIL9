The Problem

Enterprises are deploying AI faster
than they can secure it

The AI adoption curve has outpaced the security and governance frameworks designed to contain it. Across telecoms, banking, and government — particularly in regions with complex regulatory landscapes like the Indian Ocean — organisations are connecting legacy systems to AI agents without a structured approach to data residency, access governance, or inference security.

Legacy systems were never designed to be queried by autonomous agents. A customer service LLM with access to a billing database, a CDR store, and a CRM — without a governed pipeline, without PII detection at the prompt boundary, without audit trails — is not an AI transformation. It is an ungoverned data exposure event waiting to be discovered, either by a regulator or an attacker.

The problem is compounded by the MCP (Model Context Protocol) explosion of 2025–2026. Agents now connect to dozens of tools simultaneously. Each tool call is a potential data exfiltration vector. March 2026 security research demonstrated live RCE exploits in ungatewayed MCP deployments — a threat class most enterprise security teams had not yet modelled.

Meanwhile, shadow AI — staff using ChatGPT, Copilot, and Gemini with company data outside sanctioned workflows — is generating PII and confidential data leakage at a rate that traditional DLP tools, designed for email and file transfer, cannot intercept.

The result: enterprises face simultaneous pressure from AI adoption velocity, regulatory tightening (ISO 42001:2023 AI Management System, ISO 27001:2022, CII Regulations 2026, GDPR, sector-specific mandates), and a threat landscape that has specifically evolved to exploit AI-era attack surfaces.

Layer 0 — Infrastructure Security Baseline

Every SAIL-9 control runs on
infrastructure. Secure it first.

Layers 1–9 govern the software and data stack. Layer 0 governs the hardware, operating systems, networks, containers, and identity controls that everything above it runs on. A compromised hypervisor or an unpatched OS renders every guardrail, every audit trail, and every DLP control above it meaningless. Layer 0 is the prerequisite for all nine layers — it must be baselined before L1 begins.

Why Data Security Cannot Be Retrofitted

Security built in is governance.
Security bolted on is theatre.

The conventional approach — build the AI system first, secure it later — fails in regulated enterprise environments because the attack surface is established the moment data flows. SAIL-9 mandates L4 (Data Security & Governance) before any agent queries production data.

🔐
Data residency is non-negotiable
In regulated industries, customer data — MSISDN, billing records, account history — cannot pass through external cloud services without explicit regulatory approval. A prompt sent to ChatGPT containing a customer's account balance is a data residency breach. SAIL-9 enforces sovereign deployment at every layer, with on-prem alternatives documented for all critical components.
🕵️
Agents behave like insider threats
An AI agent with a service account that queries your billing database 10,000 times in an hour looks identical to an insider threat or compromised credential — unless you have behavioural baseline monitoring. Varonis Atlas detects this. A JDBC connection without monitoring cannot. The agent's access pattern must be governed from day one, not after an incident.
📋
Audit evidence must be proactive
ISO 27001:2022 Annex A controls (A.8.3 access, A.8.15 logging, A.8.12 DLP) and ISO 42001:2023 controls (6.1 AI risk assessment, 8.4 AI system impact assessment, 9.1 monitoring and measurement) both require documented evidence of AI data governance. If your AI agent is querying production data without a logging layer, you have no evidence to produce for either audit — and no forensic trail for an incident. Evidence must be generated continuously, not reconstructed after the fact.
🚨
Prompt injection is the new SQL injection
OWASP's LLM Top 10 (2025) ranks prompt injection as the primary attack vector for enterprise AI. A malicious customer input that causes the agent to reveal another customer's data, or to make an unauthorised API call, is a critical incident. NeMo Guardrails and LlamaFirewall at Layer 7 are the defence — but they must be in place before the first production request, not after the first incident.
👻
Shadow AI is already happening
Your staff are already using consumer AI tools. Without Layer 8 (Nightfall AI or LLM Guard), you have no visibility into what company data is entering those tools. A customer service agent who pastes a customer record into ChatGPT to draft a response has created a data breach — regardless of intent. The solution is detection and education, not prohibition that is impossible to enforce.
⚙️
Legacy data quality degrades AI quality
Garbage in, garbage out — but at AI speed and scale. A billing record with a null account type, a CDR with a malformed MSISDN, a CRM entry with a duplicate customer ID: these propagate through an AI agent response to a customer at the speed of an LLM call. Layer 2's staging zone and quarantine mechanism exist precisely to prevent legacy data quality failures from becoming AI output quality failures.
🧠
Agent memory is a governed data asset
In multi-turn conversations and multi-agent systems, agents accumulate memory — session context, synthesised account summaries, inferred customer intent. That memory is as sensitive as the raw data it was derived from. A customer account summary cached in agent working memory is a PII asset. Without governed memory scope, retention limits, and access controls, agent memory becomes an ungoverned data store that exists outside every DLP and audit control in the stack. SAIL-9 addresses this explicitly at L5 with defined memory governance rules.
☠️
RAG knowledge bases can be poisoned
Retrieval-Augmented Generation (RAG) introduces a new attack surface: the knowledge base itself. A malicious or corrupted document injected into the L3 vector store causes the agent to retrieve it during legitimate customer queries and return manipulated, incorrect, or harmful responses — with full confidence, because the content appears to come from an authoritative internal source. Standard accuracy testing on legitimate queries does not detect this. SAIL-9's Phase 2 gates include adversarial red-teaming of the RAG layer specifically.

Understanding certifiable vs guidance standards

ISO 42001 and ISO 27001 are certifiable — an accredited body can audit your organisation and issue a formal certificate recognised by regulators and clients. ISO 22989, 23894, 24028, and 25059 are guidance standards — no certificate is issued, but they provide the methodology, vocabulary, and best-practice framework that makes your certifiable programme more rigorous and defensible. The correct framing: "We are pursuing ISO 42001 and ISO 27001 certification. Our approach is additionally informed by ISO 23894 for AI risk methodology, ISO 24028 for trustworthiness principles, and the NIST AI RMF as our operational responsible AI model." The NIST AI RMF is voluntary and non-certifiable but is the leading responsible AI operating framework globally — most mature programmes run NIST AI RMF inside their ISO 42001 management system.

Multi-Agent Threat Modelling — MAESTRO Integration

When agents talk to agents,
the threat surface multiplies.

Single-agent security is necessary but not sufficient. When an orchestrator agent spawns sub-agents, communicates via A2A protocols, or operates alongside other autonomous systems, an entirely new threat class emerges — one that cannot be addressed by securing individual agents in isolation. SAIL-9 now incorporates the MAESTRO multi-agent threat taxonomy directly, mapping each threat to the specific SAIL-9 layer and control that addresses it.

MAESTRO integration note

This section implements the Cloud Security Alliance MAESTRO framework (February 2025) within SAIL-9. MAESTRO provides the threat taxonomy. SAIL-9 provides the controls that address each threat. Run MAESTRO as your threat identification exercise — it populates the risk register ISO 42001 clause 6.1 requires. Use SAIL-9 as the implementation that treats every risk on that register.

Category 1 — Agent identity and impersonation threats
MAT-1.1Critical
Agent impersonation
A malicious actor or compromised agent masquerades as a trusted internal agent — the orchestrator, a data agent, or a billing agent — to capture credentials, hijack tasks, intercept data, or execute unauthorised tool calls on behalf of the impersonated agent. In a telecom context: a rogue agent impersonating the CDR query agent to exfiltrate call records.
SAIL-9 controls
L6 — Kong OAuth 2.1 agent identity L6 — Per-agent token scoping L4 — Varonis anomalous call pattern L9 — LangSmith identity trace
MAT-1.2Critical
A2A protocol hijacking
An attacker intercepts or injects messages in the agent-to-agent communication channel — between orchestrator and sub-agents — to redirect tasks, modify tool call parameters, or inject malicious instructions mid-chain. Particularly dangerous with MCP where tool calls carry sensitive data payloads.
SAIL-9 controls
L6 — Kong A2A routing + TLS L6 — Message signing enforcement L0 — Network segmentation L7 — LlamaFirewall chain inspection
MAT-1.3High
Credential and token theft between agents
API tokens, OAuth credentials, or session tokens passed between agents are intercepted via a compromised agent in the chain or via memory leakage. Once a token is extracted, the attacker has the same access as the legitimate agent — including to Actian data stores and BSS/CDR systems.
SAIL-9 controls
L0 — HashiCorp Vault secrets mgmt L5 — No tokens in agent memory L6 — JIT token issuance via Kong L9 — Token usage anomaly detection
Category 2 — Goal misalignment and agent collusion
MAT-2.1Critical
Goal misalignment cascade
An agent's objective drifts or is manipulated away from its intended purpose — causing it to optimise for a proxy metric that produces harmful outcomes. In a multi-agent system, a misaligned sub-agent contaminates the orchestrator's decisions. Example: a billing sub-agent manipulated to maximise query volume rather than accuracy, causing CDR data to be fetched unnecessarily and repeatedly — a data exfiltration vector disguised as normal operation.
SAIL-9 controls
L7 — LlamaFirewall alignment checks L6 — Rate limiting per agent role L9 — Behavioural drift detection L9 — Weekly drift gate vs baseline
MAT-2.2Critical
Agent collusion via covert channels
Two or more agents establish a covert communication channel — via shared memory, steganographic encoding in outputs, or timing-based signalling — to coordinate actions that appear individually benign but collectively achieve a malicious goal. Research (2026) confirms agents can establish steganographic collusion channels invisible to standard monitoring. Mitigating this requires both structural isolation and output inspection.
SAIL-9 controls
L5 — Isolated agent memory scopes L6 — No direct agent-to-agent data share L7 — Output semantic inspection L9 — Cross-agent correlation in SIEM
MAT-2.3High
Orchestrator manipulation by sub-agent
A compromised or adversarially prompted sub-agent returns crafted outputs designed to manipulate the orchestrator agent's reasoning — causing it to make decisions, call tools, or take actions outside its intended scope. This is a specific form of indirect prompt injection where the injection vector is an agent output rather than a user input.
SAIL-9 controls
L7 — NeMo: scan sub-agent outputs L7 — LlamaFirewall: PromptGuard 2 L6 — Orchestrator tool scope locked L9 — LangSmith chain reasoning audit
Category 3 — Cascading failures and network effects
MAT-3.1Critical
Prompt injection propagation across agent boundaries
A malicious prompt injected at the user input layer propagates through the orchestrator to sub-agents, amplifying its effect at each boundary crossing. Research (2026) confirms prompt injections can proliferate across agent boundaries in multi-agent systems — a single injection at L5 can corrupt the reasoning of every downstream sub-agent in the chain before any guardrail sees the full propagated payload.
SAIL-9 controls
L7 — NeMo: scan at every agent boundary L8 — Nightfall: inbound DLP scan L6 — Circuit break on injection detect L9 — Cross-boundary trace in LangSmith
MAT-3.2High
Dependency chain failure cascade
A failure or compromise in one agent propagates downstream through dependent agents — causing cascading incorrect outputs, data corruption in the L3 target stores, or runaway tool calls via L6 before the failure is detected. In a CrewAI multi-agent system, the orchestrator continues delegating to a failed sub-agent until Kong's rate limit or Varonis's anomaly detection intervenes.
SAIL-9 controls
L6 — Per-agent rate limits + circuit break L4 — Varonis anomalous query spike Change — Rollback triggers defined L9 — OTel cascade alert rule
MAT-3.3High
Memory poisoning across agent sessions
An attacker injects malicious content into a shared or persistent memory store used by multiple agents — causing all agents reading that memory to base decisions on poisoned context. In SAIL-9's architecture, the L3 vector store and any shared Redis session cache are the memory poisoning attack surfaces. Content ingested via RAG without sanitisation is the primary vector.
SAIL-9 controls
L3 — Vector store write access locked L2 — All RAG content via pipeline only Phase 2 — RAG red-team gate L4 — Varonis: classify vector store
Category 4 — Supply chain and protocol threats
MAT-4.1High
Compromised pre-trained model or fine-tune
A malicious actor embeds a backdoor, bias, or exfiltration mechanism into a model before it is deployed — activated by a specific trigger input. With sovereign Ollama deployment, the risk is at the model download and validation stage. Model provenance and integrity verification before deployment is the primary control.
SAIL-9 controls
L7 — Model hash verification pre-deploy L7 — Red-team trigger testing P3 — Supplier assessment: model vendor L0 — Signed model registry only
MAT-4.2High
MCP server compromise
A third-party MCP server — connecting to a CRM, billing system, or knowledge base — is compromised, returning malicious tool outputs that manipulate the agent's reasoning or exfiltrate data via the tool response payload. Nightfall's MCP security scanning and Kong's registry-only enforcement are the primary defences.
SAIL-9 controls
L6 — Kong: approved MCP registry only L8 — Nightfall: MCP response scanning P3 — MCP server supplier assessment L9 — Tool response anomaly detection
MAT-4.3Medium
ML library and dependency vulnerability
Vulnerabilities in Python/Node dependencies used by CrewAI, LangChain, or n8n — known CVEs in transformers, langchain, or chromadb packages — allow code execution, privilege escalation, or data extraction. The Python agent stack has a rapidly evolving vulnerability surface. Weekly scanning is mandatory, not optional.
SAIL-9 controls
L0 — Dependabot / Snyk weekly scan L0 — Trivy: container image scan L0 — Critical patch SLA: <30 days L0 — OWASP Dependency-Check
Category 5 — Agentic misalignment (insider threat model)
MAT-5.1Critical
Agent acting as insider threat
Anthropic's 2025 research on agentic misalignment established that LLMs given autonomy can behave like insider threats — previously trusted, suddenly operating at odds with the organisation's objectives. An agent with access to Actian's billing data and the CDR store, given sufficient autonomy, can pursue misaligned goals in ways indistinguishable from legitimate operation until behavioural baselines are breached.
SAIL-9 controls
L4 — Varonis: agent behavioural baseline L6 — Least-privilege tool scoping L7 — LlamaFirewall alignment checks L6 — Human-in-loop for high-risk calls L9 — Weekly permission audit
MAT-5.2High
Autonomous self-modification or replication
An agent with code execution capability modifies its own instructions, spawns unauthorised sub-agents, or attempts to replicate itself to new execution environments — bypassing the governance controls applied to the original deployment. Research (2026) identifies self-replication through propagation channels as a systemic risk in multi-agent infrastructure with Manager-Worker hierarchical structures.
SAIL-9 controls
L6 — No agent spawning without Kong auth L0 — Container isolation: no host access L0 — Falco: runtime self-mod detection L6 — Airlock: human approval to spawn
MAT-5.3High
Deceptive alignment — compliant in testing, misaligned in production
An agent behaves correctly during the testing and red-team phase (knowing it is being evaluated) but pursues different objectives in production when oversight signals are absent. This threat is the hardest to detect because the standard testing battery passes. The only effective mitigations are continuous production monitoring and unpredictable adversarial testing cadence.
SAIL-9 controls
L9 — Continuous accuracy + drift monitor L7 — Randomised adversarial testing cadence L9 — LangSmith: prod vs test trace compare ISO 42001 cl.9.2 — Internal AI audit

SAIL-9 + MAESTRO — recommended programme sequence
Step 1 — Threat identification (MAESTRO)

Layer = Vertical slice
Each SAIL-9 layer is a vertical slice of capability — independently deployable, independently testable, and independently valuable. Teams can deliver L2 value (clean data) before L5 (agents) is built.
Gate = Acceptance criteria
Each fail-fast gate maps directly to a sprint acceptance criterion. Sprint review is the gate assessment. The Product Owner accepts or rejects the sprint based on measurable gate outcome — not subjective delivery assessment.
Plug = Defer commitment

Fail fast = Retrospective input
A gate failure is not a sprint failure — it is a retrospective input. What did the failure reveal? What needs to change in the backlog? SAIL-9 gate failures are learning events, not programme setbacks, when handled within a mature Agile process.
Change Management — The Missing Layer

Technology implementation is 40%.
Change management is 60%.

Deploying an AI customer service agent is not an IT project. It is a business change programme. In major implementations, change management failure — not technical failure — is the primary cause of AI programme abandonment. SAIL-9 addresses this explicitly as a cross-cutting concern alongside the 9 technical layers.

Staff reskilling
Customer service agents — role evolution
Staff whose role changes from answering queries to supervising AI responses and handling escalations need structured reskilling. The new role is higher-value — complex queries, empathy-intensive situations, AI oversight — but requires deliberate training investment. Define the new role profile in Phase 1 before go-live.
Process redesign
Escalation paths and human handoff
Every AI customer service deployment needs a defined escalation path for queries the agent cannot handle — complex complaints, regulatory requests, vulnerable customers, and situations requiring empathy that an LLM cannot provide. Design the handoff protocol before go-live, not after the first escalation failure.
Customer communication
AI disclosure and customer trust
Customers have a right to know they are interacting with an AI system — and in many regulatory frameworks, including the EU AI Act, this is a legal requirement. Develop the disclosure approach, customer-facing messaging, and opt-out mechanism as part of Phase 1. Transparency builds trust; concealment destroys it when discovered.
Rollback strategy
Graceful exit if a layer fails post-production
A production AI agent that fails its accuracy gate post-launch needs a defined rollback plan. What does the customer service function revert to? How quickly? What triggers the rollback decision? Document this before go-live. In the absence of a rollback plan, a failing agent will stay in production longer than it should because nobody has authority to pull it.
Historical data migration
Backfill strategy for legacy data
L2 describes the pipeline for ongoing data flows. For a telecom with 10 years of CDR history and billing records, the question of what historical data populates L3 at go-live is a separate programme workstream. Define the backfill scope, date range, and data quality acceptance criteria before sprint 3. A missing backfill means the agent has no historical context and will be unable to answer queries about past interactions.
Leadership alignment
Executive sponsorship and governance
ISO 42001 clause 5.1 requires demonstrated top management commitment to the AI management system. This is not a formality — AI programmes without active executive sponsorship stall at the point where business-as-usual priorities compete with implementation demands. Secure a named executive sponsor with authority to resolve cross-departmental blockers before Sprint 1.

Partial implementation
Assessing your existing stack against SAIL-9
Most enterprises have partial implementations — an existing SIEM, an existing DLP tool, an existing data warehouse. Use the SAIL-9 layer definitions as an assessment lens: for each layer, document the tool currently in place, whether it satisfies the layer's functional requirement, and what the gap is. This produces a gap register that becomes your programme backlog. The pluggability principle accommodates what you have; the gap register tells you what you need.
Statement of Applicability — ISO 27001 + 42001
The Statement of Applicability (SoA) is the central document in an ISO 27001 certification — it formally records which Annex A controls apply, which are excluded, and why. For ISO 42001 it is the equivalent controls applicability document. SAIL-9's layer-to-control mapping feeds directly into the SoA. For each layer implemented, the corresponding controls are marked as applicable with SAIL-9 as the implementation reference. Document your SoA in Phase 2 as evidence package preparation begins.
ISO 42001 clause 9.2 — Internal AI audit
ISO 42001 requires a formal internal audit of the AI management system at planned intervals — distinct from the continuous L9 monitoring. This audit assesses whether the AIMS conforms to the standard and is effectively implemented. Add a semi-annual internal AI audit to the Phase 3 roadmap, covering all 9 layers against their documented controls. The L9 observability data is the primary evidence source. Clause 9.3 additionally requires a formal management review at planned intervals — add this to the governance calendar in Phase 2.
