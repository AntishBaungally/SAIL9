Vector RAG vs. Vectorless RAG
— and the right balance for enterprise

Embedding similarity and structured retrieval answer different questions over the same data. A defensible enterprise architecture rarely chooses one — it combines both, phased to match the audit and compliance demands of regulated environments.

"Vector RAG" and "vectorless RAG" are often framed as competing approaches, but in practice they answer different questions over the same data. The right enterprise architecture rarely chooses one outright — it combines both, with the balance determined by the type of content and the type of query.

1. The core comparison

Before deciding on an architecture, it helps to separate what each approach is actually good at — and where each one quietly breaks down in production.

Dimension Vector RAG Vectorless RAG
Best at Semantic / fuzzy matching — finding conceptually related content even with different wording Exact, structured, or relational matching — IDs, clause numbers, multi-hop relationships
Mechanism Embedding similarity search (dense vectors) Lexical search (BM25), metadata filters, SQL, or knowledge-graph traversal
Auditability Lower — similarity scores are not easily explainable to an auditor Higher — retrieval path is traceable and defensible ("why did you say this?")
Weak spot Exact-match queries (codes, dates, regulation numbers); chunking and embedding drift Paraphrase / synonym handling; requires upfront structuring effort
Ideal content Narrative text — policy documents, BIA write-ups, meeting notes Structured records — ISO control mappings, regulation clauses, system inventories

2. Vector RAG

Documents are split into chunks, each chunk is converted into a dense numeric embedding, and queries are matched by similarity in vector space. Tooling is mature — Qdrant, Weaviate, Milvus, Pinecone, and pgvector all support this well.

Strengths

  • Captures meaning, not just words — finds related content even when terminology differs
  • Scales well to large, unstructured corpora
  • Mature, well-supported ecosystem with strong tooling

Weaknesses

  • Embedding quality is a black box — hard to explain why a result was retrieved
  • Struggles with exact-match queries: IDs, dates, numeric thresholds, clause references
  • Chunking strategy heavily affects retrieval quality
  • Re-embedding required when the underlying model is upgraded
  • "Lost in the middle" effects with large retrieved contexts

3. Vectorless RAG

Retrieval relies on lexical search (BM25), structured metadata filters, SQL-style queries, or knowledge-graph traversal over explicitly modelled entities and relationships — no embeddings involved.

Strengths

  • Precise exact-match and entity-level retrieval — ideal for ISO control IDs, regulatory citations, system names
  • Interpretable and debuggable — retrieval logic can be shown step by step
  • No embedding drift; stable across model upgrades
  • Multi-hop relational reasoning (graph traversal) handles questions a single similarity search cannot

Weaknesses

  • Weaker on paraphrase and synonym handling
  • Knowledge-graph approaches require upfront modelling and maintenance effort
  • Pure keyword search misses conceptually related but differently worded content

4. A common middle ground: structured JSON inside a vector database

Most modern vector databases — Qdrant, Weaviate — support storing structured JSON as payload/metadata alongside, or instead of, an embedding. This gives rise to two distinct patterns.

Pattern 1

Vector DB as a document store only

Structured JSON (control IDs, BIA records, system metadata) is stored as payload, and retrieval happens purely via metadata filters — exact match on control_id, regulation_ref, system_name. No embedding similarity is used. This is effectively vectorless retrieval using a vector database as the storage layer — functionally equivalent to a regular database (Postgres / MongoDB), often simpler to operate if you're already using one for vector workloads.

Pattern 2 · Recommended default

Hybrid — embedding + structured payload

Each record gets both a natural-language summary, embedded for semantic discovery (e.g. "ISO 27001 A.5.7 Threat Intelligence — owner: Eddy — status: implemented"), and the full structured JSON as metadata, used by the agent for precise reasoning once the record is retrieved. A single query surfaces relevant records semantically, while the agent receives exact structured data — no separate graph database required for simple, one-hop relationships.

5. How hybrid retrieval actually works

"Query both and get the best response" requires a defined fusion strategy — it does not happen automatically.

01
Query received — e.g. "What's our exposure on the CII compliance deadline for OCS?"
02
Vector path — embed query, run similarity search, return semantically related narrative chunks
03
Structured path — extract entities/filters (system=OCS, topic=CII, year=2026), run exact metadata match, return precise records
04
Fusion — merge and rerank both result sets
05
Merged context passed to the LLM for the final answer

Fusion strategies

Approach Description
Reciprocal Rank Fusion (RRF) Combines rankings from both result lists without requiring comparable scores. Simple and natively supported (Qdrant, Weaviate, OpenSearch).
Weighted scoring e.g. 0.6 × vector_score + 0.4 × keyword_match_score. Requires tuning per use case.
LLM-as-reranker Both result sets are passed to the LLM, which judges relevance directly. More expensive, more accurate, fits agentic orchestration.

6. How enterprises are doing this (2025–2026)

Layer 1 — Hybrid search is now the baseline

Pinecone, Weaviate, Qdrant, Elasticsearch/OpenSearch, and Azure AI Search all ship dense + sparse hybrid search with RRF built in. Pure vector-only RAG is rarely used in production enterprise knowledge bases — hybrid is table stakes, not an advanced feature.

Layer 2 — Metadata filtering as the de facto access control layer

Most production "RAG" is really filtered hybrid search — retrieval is scoped by tenant, document type, date range, and access permissions (RBAC) before similarity runs at all. In regulated industries, this filtering layer is often the actual compliance control, ensuring the LLM cannot retrieve documents a user is not authorised to see.

Layer 3 — Knowledge graphs for compliance and GRC use cases

For audit-defensible, multi-hop reasoning — control → regulation → system → owner — enterprises increasingly layer a graph (e.g. Neo4j) on top of vector search over unstructured policy text. Microsoft's GraphRAG was built specifically because pure vector RAG fails on "global" questions that require reasoning across an entire corpus.

Layer 4 — Agentic retrieval (current frontier)

Rather than a fixed pipeline, an orchestrator agent chooses the retrieval strategy per query — SQL, vector, graph, or a combination — and can self-correct if results are thin. LangGraph, CrewAI, and most current agent frameworks recommend this pattern for complex enterprise knowledge tasks, at the cost of higher per-query expense.

7. Recommended balance for a GRC / compliance stack

For a compliance-heavy domain — ISO 27001 / 22301 controls, regulatory citations, BIA records, system inventories — the recommended phased approach is:

01
Phase 1
Hybrid vector + keyword search in a single vector DB (e.g. Qdrant) with metadata filters for access control. Covers roughly 80% of retrieval needs with minimal complexity.
02
Phase 2
Add a lightweight knowledge graph specifically for compliance entity relationships — control → framework → system → owner → regulation. Even a few hundred nodes provides multi-hop traceability that vector metadata filters cannot.
03
Phase 3
Introduce agentic orchestration (e.g. via CrewAI) where an orchestrator agent selects vector search, structured lookup, or graph traversal per query — justified once query variety and corpus size grow beyond what a single hybrid query reliably covers.
Key principle

Don't build all four layers at once. Hybrid search plus metadata filtering delivers most of the value early. The graph layer is the differentiator specifically where audit-defensible, multi-hop traceability matters — which is precisely where pure vector similarity scores cannot withstand scrutiny.
